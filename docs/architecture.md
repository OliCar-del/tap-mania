# Architecture and Code Execution Summary

## 1. System Architecture Overview

The firmware is designed for an **STM32L433** microcontroller (configured via STM32CubeMX/IDE and built using a CMake workflow). It functions as a real-time embedded system that bridges low-latency hardware peripherals (timers, audio generation, button interrupts, and addressable LEDs) with a file system layer for standalone game execution, alongside a USB serial command link for configuration management.

```
                  +---------------------------------------+
```

## 2. Code Modules & Component Breakdown

### Core Game Engine & Parsing

* **`tsq_parser.c` / `tsq_parser.h**`
Responsible for streaming and parsing the custom `.tsq` rhythm sequence text files from the filesystem. It parses the metadata header (Name, Artist, BPM, Difficulty, and Beat Offset) and sequentially processes bars of row strings (`0`, `1`, `2`) to interpret taps, multi-presses, and hold notes.
* **`game_state.c` / `game_state.h**`
Manages the primary operational state machine of the machine (e.g., `MENU`, `SONG_SELECT`, `PLAYING`, `RESULTS_SCREEN`). It handles transitions, restarts, and game over sequences.
* **`scoring.c` / `scoring.h**`
Processes timing parameters down to the millisecond. When a hardware input event triggers, this module calculates the delta between the ideal beat-time and the actual player button press, mapping it to specific accuracy classifications (*Perfect!*, *Good!*, *OK*, *Poor*, *Miss*) and updates the total score and combo counters.

### Hardware Interface & Drivers

* **`audio.c` / `audio.h**`
Controls the monoaural audio stream. It interacts with the audio subsystem (likely utilizing the Serial Audio Interface — SAI or DAC via DMA) to read `.wav` data structures from the storage media, handling the configured audio latency offsets to match physical beats perfectly.
* **`ARGB.c` / `ARGB.h` & `led_display.c` / `led_control.c**`
Provides an optimized low-level driver for the addressable 16x4 WS2812B RGB LED matrix. It converts structural step paths into timed NRZ serial signals (often over SPI or TIM+DMA) to animate oncoming steps down the lane while dynamically lighting up the internal button LEDs.
* **`SH1106.c` / `SH1106.h` & `fonts.c**`
A hardware display driver targeting the I2C/SPI based SH1106 OLED module. It handles character arrays and coordinate plotting for system menus, song selections, and operational statistics.
* **`button_handler.c`, `button_debounce.c`, `config_input.c**`
Handles the asynchronous input processing. Pin interrupts generated by the four primary illuminated arcade switches and auxiliary controls are debounced using soft timers before being dispatched to the step engine.

### Subsystems & Connectivity

* **`sd_card_utils.c` / `sd_card_detect.c**`
Integrates the **FatFs** middleware platform directly above hardware SPI/SDMMC layers to interface reliably with external microSD cards containing song packs.
* **`config_storage.c`**
Implements a persistent storage system to save personalized device configurations (like customized user timing windows between 1ms and 500ms) within the STM32's internal non-volatile Flash memory pages.
* **`command_handler.c` / `uart_debug.c**`
Processes incoming serial streams passing through the USB interface (interfacing with the external USB microcontroller bridge). It maps PC-side adjustments into executable variables, allowing communication with the companion configuration GUI application.

## 3. Firmware Execution Flow

The program life cycle maps into three operational phases:

### Phase A: Initialization

1. **System Init:** Hardware abstraction layers (`HAL_Init`) configure internal phase-locked loops (PLLs) to supply stable clock lines across communication lines and compute registers.
2. **Peripheral Init:** Resets and schedules interfaces for internal systems (FatFs for SD card access, I2C/SPI for the SH1106 OLED, Timer channels for PWM/DMA LED patterns, and sound peripherals).
3. **Config Extraction:** Reads the internal Flash non-volatile space to extract customized game latency parameters. If blank, it initializes system default timings.
4. **Storage Verification:** Attempts to mount the microSD interface. If the card is missing or corrupted, a critical alert prints directly to the display loop.

### Phase B: Game Loop Execution

Once running, the main loop continually tracks operational events:

* **State Control:** Monitors current selections. If navigating menus, it decodes button ticks to cycle through parsed song variables.
* **Rhythm Mode Processing:** Upon starting a track, the processor synchronizes audio tracks with a master tick timer. The engine matches the song's BPM to periodically step row data forward, moving rows across the 16x4 matrix.
* **The Beat LED Window:** A software event schedules the hardware Beat LED to flash precisely `50ms` before the beat hits, cutting power `50ms` afterward.
* **Input Capture:** If a player triggers a button, an interrupt logs a timestamp. The delta between this click and the current track timing index is measured by `scoring.c` to assign score weightings instantly.

### Phase C: Communication & Interventions

* **UART Intercepts:** Asynchronous USB data packets from the PC GUI drop into ring buffers. `command_handler.c` executes these actions out of the game's critical loop (e.g., retrieving configuration lists or committing new timing windows directly to Flash).
